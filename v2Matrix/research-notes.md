### ElementsFactory landscape

There are **multiple** ElementsFactory interfaces — this is a known trap
that burned the Block Builder work. For SysMLv2 creation:

- `com.dassault_systemes.modeler.sysml.model.ElementsFactory`
  *(extends `ElementsFactory, SysMLElementsFactory`)* — the SysML one.
- `com.dassault_systemes.modeler.kerml.model.ElementsFactory` — KerML
  base (inherited by the SysML factory).
- `com.nomagic.uml2.impl.ElementsFactory` — the legacy UML2 factory.
  `project.getElementsFactory()` returns an object compatible with this
  interface, but **we should confirm per creation site** that the
  returned factory can produce SysMLv2 metaclasses. If it can't, we cast
  or obtain a SysML-specific factory from the project profile.

**Iteration 1 open question:** does `project.getElementsFactory()`
produce an instance that supports `createRequirementUsage()`? First code
we write will probe this with a `respondsTo`/`instanceof` check; if not,
we route through the SysML factory lookup (TBD).

## Key method signatures

### `SatisfyRequirementUsage` (interface)

> "A SatisfyRequirementUsage is an AssertConstraintUsage that asserts,
> by default, that a satisfied RequirementUsage is true."

```java
RequirementUsage getSatisfiedRequirement()
Feature          getSatisfyingFeature()
```

This is the full API for driving FR-1 / FR-6. For a satisfy cell to
exist, we read one `SatisfyRequirementUsage` and call both getters —
row is `satisfyingFeature`, col is `satisfiedRequirement` (or vice
versa after FR-2 swap).

### `RequirementUsage` (interface)

Key methods for FR-7 and constraints navigation:

```java
List<ConstraintUsage> getAssumedConstraint()
List<ConstraintUsage> getRequiredConstraint()
Usage                 getSubjectParameter()   // FR-7 subject-based derived kind
```

### `ViewUsage` (interface) — FR-15 persistence target

```java
List<Element>         getExposedElement()
List<ViewpointUsage>  getSatisfiedViewpoint()
List<Expression>      getViewCondition()
ViewDefinition        getViewDefinition()
RenderingUsage        getViewRendering()
boolean               includeAsExposed(Element element)
```

For FR-15 we attach string configuration via `FeatureValue` + `LiteralString`:

```java
// FeatureValue
void             setFeatureWithValue(Feature f)
void             setValue(Expression v)

// LiteralString  (subtype of Expression)
void             setValue(String v)
```

So the pattern for saving a config key is: create a Feature on the
ViewUsage, create a FeatureValue bound to that Feature, create a
LiteralString as the value, wire them together inside one
`SessionManager` session.

### `LibraryPackage` (interface) — FR-4 scope heuristic

```java
boolean   isStandard()
Namespace libraryNamespace()
void      setIsStandard(boolean value)
```

**FR-4 default-scope rule**: walk an element up through its owners;
if any owner is a `LibraryPackage` with `isStandard() == true`, the
element is a standard-library element and excluded from the default
scope. See `scripts/v2Matrix/LibraryDetector.groovy`.

### `Dependency` (interface)

```java
List<Element> getClient()
List<Element> getSupplier()
```

Plain UML-style client/supplier. For FR-5 when kind = `Dependency`,
client is the row element and supplier is the column.

## Traversal patterns

No SysMLv2-specific finder exists; we reuse Cameo's legacy helpers:

- `com.nomagic.magicdraw.uml.Finder.byTypeRecursively().find(owner, types, includeOwner)`
  — walks the Element containment tree and collects all instances of
  the given types. Works because SysMLv2 Usage types are KerML
  Elements, and `getOwnedElement()` is defined on KerML `Element`.

For filtering by scope (FR-4):

```groovy
Collection<Element> collectInScope(Namespace scope, Class<?>[] types) {
    Finder.byTypeRecursively().find(scope, types, false)
        .findAll { elem -> !LibraryDetector.isInStandardLibrary(elem) }
}
```

## Classpath validation limitation (iteration 0 discovery)

**`groovyc` can't resolve SysMLv2 classes from any filesystem classpath.**

We tried pointing the validator at `lib/*` and at
`plugins/com.nomagic.magicdraw.sysml2.plugin/lib/*`; both produced
`unable to resolve class` errors for `…kerml.Element` and
`…kerml.LibraryPackage`. A full scan of every `.jar` under the install
found no `.class` file at the expected path.

Conclusion: SysMLv2 / KerML classes are **generated or loaded
dynamically at Cameo startup** (EMF-style), not shipped as precompiled
bytecode. This matches the user's earlier insight that Cameo scripts
must run inside the application, not against its classpath.

**Implication for our toolchain:**

- `validate_script_syntax` will always return compile **errors** for
  SysMLv2 imports when run outside Cameo. Those errors are not bugs.
- The MCP's **automatic FQN cross-check** (`lintWarnings`) *is* the
  authoritative guardrail against hallucinated imports. It consults
  the shipped Javadoc, which does contain the SysMLv2 classes.
- For each iteration going forward, the acceptance criterion on
  `validate_script_syntax` is: `lintWarnings` is empty. Compile
  errors mentioning `com.dassault_systemes.*` / `kerml.*` /
  `sysml.model.sysml.*` are expected and ignored.
- The runtime test is always "load the script in Cameo's Groovy
  console (or via the REST harness) and watch `logs/v2Matrix.log`".

This asymmetry is a real limitation, not a bug — it's why the agent
protocol requires `lintWarnings` to pass before a commit, rather than
relying on `ok: true` alone.

## Iteration 1 API discoveries (probe scripts, SysMLv2 element creation)

### Factory access
- `project.getElementsFactory()` returns `com.nomagic.uml2.SmartElementsFactory` — **NOT** the SysML factory.
- `com.dassault_systemes.modeler.sysml.model.sysml.SysMLFactory.eINSTANCE` is the correct singleton.
  Returns `SysMLFactoryImpl` with 140+ `create*()` methods.
- `com.dassault_systemes.modeler.kerml.model.kerml.KerMLFactory.eINSTANCE` works for KerML base elements.
  `kermlFactory.createPackage()` works. `sysmlFactory.createPackage()` does **not exist**.
- KerML and UML2 Element hierarchies are **separate** — `ModelElementsManager.addElement()` (which takes `mdkernel.Element`) cannot accept SysMLv2 elements.

### Naming
- `element.setDeclaredName(String)` — from `kerml.Element`; this is the naming API.
- `element.getHumanName()` reflects the declared name: returns `"RequirementUsage TestR1"` after `setDeclaredName("TestR1")`.
- `element.setName()` does **not** exist on SysMLv2 elements.

### Ownership
- `element.setOwner(namespace)` works inside a `SessionManager` session.
- To make P6 own Q: `q.setOwner(p6)` — Q becomes a sub-feature of P6.

### SatisfyRequirementUsage — derivation caveats (iteration 1 runtime tests)
- **No `setSatisfiedRequirement()` setter exists.** `getSatisfiedRequirement()` is derived.
- **No `setSatisfyingFeature()` setter exists.** `getSatisfyingFeature()` is derived.
- **Runtime test results** with a fixture created programmatically:
  ```
  satisfy.getOwner()                  → P1        (correct — we called setOwner)
  satisfy.getSatisfyingFeature()      → null      (unexpected — derivation returns null)
  satisfy.getSatisfiedRequirement()   → satisfy   (WRONG — returns self as fallback)
  satisfy.getOwnedTyping()            → [FT: type=R1]  (correct — our FeatureTyping is registered)
  satisfy.getType()                   → [RequirementCheck]  (doesn't include R1)
  ```
- **Conclusion**: Cameo's implementation of `getSatisfiedRequirement()` does NOT read from our programmatically-added `FeatureTyping.type`. It likely expects a different wiring (possibly `ReferenceSubsetting`, or direct EMF reference via an internal EReference). For user-authored SysMLv2 written as `satisfy req R1 by P1`, Cameo's textual compiler uses that different wiring.
- **Matrix-code contract (MatrixModel.groovy)**: use a two-step fallback when querying a SatisfyRequirementUsage:
  ```groovy
  RequirementUsage resolveSatisfiedRequirement(SatisfyRequirementUsage s) {
      def r = s.getSatisfiedRequirement()
      if (r != null && r != s) return r
      return s.getOwnedTyping()?.collect { it.getType() }
                              ?.find { it instanceof RequirementUsage }
  }
  Element resolveSatisfyingFeature(SatisfyRequirementUsage s) {
      def f = s.getSatisfyingFeature()
      return f != null ? f : s.getOwner()
  }
  ```
  The second fallback (`getOwner()`) matches the existing pattern in
  `scripts/RequirementSatisfyMatrixGraphics.groovy`.

### SubjectMembership (FR-7) — caveat from iteration-1 testing
- `factory.createSubjectMembership()` works; `subjM.setOwner(r7)` works.
- **`subjM.setMemberElement(p2)` TRANSFERS OWNERSHIP** of p2 out of its original container (despite what the method name suggests). Runtime evidence: TF1's `ownedMember` list lost P2 after this call. Do NOT use `setMemberElement` to reference an element owned elsewhere.
- `subjM.setOwnedSubjectParameter(p2)` also transfers ownership (method name is honest about this).
- **Correct pattern (not yet verified)**: create a fresh `ReferenceUsage` inside the `SubjectMembership`, then use a `ReferenceSubsetting` on that `ReferenceUsage` pointing to the actual subject (p2). This keeps p2 in its original container.
- For iteration 1 the fixture builder creates an **empty** SubjectMembership on R7; FR-7 is deferred until we verify the ReferenceSubsetting pattern.

### Project model root
- For DST-native SysMLv2 projects: `project.getModel()`, `project.getPrimaryModel()`, `project.getModels()` all return null/empty.
- `project.getModels()` returns `Collections.emptyList()` (immutable).
- The browser root contains `IPrimaryProject` / `ILockableProject` nodes — not KerML Namespace.
- **Correct pattern**: require the user to select a named user Package in the browser before running a creation script. This is the same pattern as `RequirementSatisfyMatrixGraphics.groovy`.
- Only accept a namespace with a non-null, non-empty `getDeclaredName()` to avoid capturing system library namespaces (which also show as Namespace instances in the tree but have null names).
